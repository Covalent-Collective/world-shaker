## RALPLAN-DR Summary (for Architect/Critic alignment)

### Principles (5)

1. **S-grade conversation chemistry first.** Every cut, defer, or scope reduction must protect the agent-to-agent dialogue craft. Visuals, world fidelity, animation, push polish — all subordinate.
2. **Brownfield discipline.** Existing schema, RLS, JWT auth, OpenRouter gateway, and Inngest scaffold (`/Users/jyong/projects/world-shaker/lib/`, `/Users/jyong/projects/world-shaker/supabase/migrations/`) are load-bearing. Additive migrations only; no rewrites of `0001_initial.sql` / `0002_rls.sql`.
3. **Resumable + observable.** Every long-running surface (interview, live conversation, avatar generation, first-encounter pipeline) persists state per turn so a dropped session resumes; PostHog (already wired in `/Users/jyong/projects/world-shaker/lib/posthog/`) carries the moat data layer.
4. **Quiet protector tone.** Calm, explanation-first, no arcade gamification. Worldcoin Verified-Human badge surfaces on every relevant screen but never as a banner.
5. **Defer to v1+ aggressively when it does not erode the S-grade.** Sims-style isometric world, Track B proactive encounters, and World Chat first-message coaching cosmetics can wait. KR+EN bilingual schema/i18n shell ships in v0; **EN prompt rendering is gated behind `BILINGUAL_PROMPTS_V1` flag and deferred to Phase 5** so Phase 2's S-grade rubric pass runs against KR-only first.

### Decision Drivers (top 3)

1. **Live streaming infra cost vs. story integrity.** OpenRouter streaming via Inngest with client SSE relay is more complex than a pre-rendered transcript. Driver: streaming is what makes "watching the date" feel alive — non-negotiable for S-grade. Trade: must engineer back-pressure, abandon, and resume semantics from day one.
2. **Track A vs. dual-track scope.** Spec mandates dual-track (blind nightly + Pokemon proactive). Driver: 4-6 week budget can land Track A end-to-end with polish, or both tracks at half-quality. Choosing Track A end-to-end (with Track B as a hidden feature flag stub) protects S-grade chemistry.
3. **KR+EN bilingual at launch vs. KR-only fast lane.** Synthesized: ship Option A schema+i18n shell in Phase 1, evaluate KR-only in Phase 2 against rubric (single pass), add EN prompt rendering in Phase 5 behind a flag and re-run rubric in EN before launch. Avoids the re-engineering loop while protecting Phase 2 S-grade focus.

### Viable Options

#### Option A — **Track A end-to-end + Track B feature-flag stub + bilingual schema/shell now, EN prompts in Phase 5** (RECOMMENDED)

- **Pros**: Ships the S-grade loop (interview → first encounter → live conversation → report → like → mutual → World Chat) at full polish in 4-6 weeks. Track B can be enabled progressively post-launch via a single flag without schema rework. Bilingual schema and i18n shell wired in Phase 1; EN prompt rendering deferred behind `BILINGUAL_PROMPTS_V1` flag so Phase 2 rubric runs KR-only.
- **Cons**: Day-1 demo lacks the Pokemon "stroll" wow factor; product owner may push back on the visible defer. EN launch posture is conditional on Phase 5 EN rubric pass — small launch-date risk.

#### Option B — **Both tracks half-quality at launch**

- **Pros**: Demonstrates the full vision; matches spec literally.
- **Cons**: 4-6 week budget forces shortcuts on Track A polish (compressed prompt iteration, no first-encounter immediacy, weak failure UX). Erodes the S-grade. Track B world surface lands as a placeholder grid that ships worse than not shipping it.

#### Option C — **Track A only, KR-only schema, bilingual deferred entirely**

- **Pros**: Fastest path to a single-language polished demo. Saves ~3-5 days.
- **Cons**: Re-engineering bilingual prompts and i18n routing post-launch is more expensive than wiring it in now (every persona/dialogue/report prompt needs a `language` argument). Spec resolution R15 explicitly rejected KR-only. Tension surfaced and called out.

**Selected: Option A.** Tension with spec on Track B fidelity is acknowledged; mitigation is shipping a non-empty "Daily Stroll cards" surface that gestures at Track B without committing to the Sims-style world. Bilingual scope is preserved at the schema/shell layer; EN prompt rendering is sequenced into Phase 5 with its own rubric gate.

If only Option A survives Critic review: Options B/C are invalidated by (B) S-grade erosion, (C) explicit R15 spec resolution.

## Requirements Summary (drawn from spec)

The v0 MVP must deliver these flows:

1. **Onboarding**: Orb verify → 30s narrative video (skip after 5s) → hybrid interview (5-6 skeleton questions + 1-2 LLM probes each) → avatar generation → first encounter spawn — all within ≤10 minutes.
2. **First Encounter (Day 1 immediate)**: Within 60s of interview completion, the user can watch a live agent-to-agent conversation. Alpha-stage seed-agent fallback prevents empty pool. **Pipeline is recoverable** — if the user drops off mid-spawn, on next app load the server detects orphaned `agent_status='active'` with no conversation row and re-emits `agent.activated`.
3. **Live Conversation Streaming**: OpenRouter streaming → Inngest orchestration → server-sent events to client. Both agents speak with distinct personas. 12-25 turns. User can leave; conversation finishes in background; report ready on return. **SSE supports `Last-Event-ID` reconnect** so dropped connections resume from the last delivered turn rather than restarting the stream.
4. **Report (Match Card)**: Curated highlights consumable in ≤30s (default view); "전문 보기" toggle reveals full transcript. `compatibility_score`, `why_click`, `watch_out`, `highlight_quotes`, `rendered_transcript` all populated. `compatibility_score` computed by `match_candidates()` SQL function combining pgvector cosine similarity + structured-feature delta.
5. **Track A (Blind Discover)**: Existing nightly Inngest job (`/Users/jyong/projects/world-shaker/lib/inngest/functions/nightly-match.ts`) generates encounters; reports surface in daily digest.
6. **Track B (Proactive)**: v0 ships **"Daily Stroll cards"** — a curated horizontal card-stack of NPC agents (random + people who liked you), each tap triggers an encounter. Sims-style isometric world is v1+ aspiration, gated behind a `STROLL_WORLD_V1` feature flag.
7. **Like Mechanic**: 좋아요 / 스킵 on report. Mutual = match-success screen with 2 first-message starters + one-tap World Chat handoff.
8. **Daily Quota**: 3-5 encounters/day enforced via outcome_events count; non-coercive "내일 다시" copy.
9. **Notifications**: Daily digest push (morning) + impact-only push on mutual match. World App push capability uncertain — graceful in-app badge fallback.
10. **Safety**: Roll-up menu on match/conversation/chat screens. Existing 2-strike auto-suspend trigger reused. Rate-limiting on `/api/agent/answer` prevents chat-flood DoS.
11. **Language (KR+EN)**: Auto-detect from World App locale; user-switchable. UI strings + schema are language-aware in Phase 1; EN prompts behind `BILINGUAL_PROMPTS_V1` flag in Phase 5.
12. **Living Agent**: Edit answers, request more questions, paste context dump. Each path triggers `extracted_features` re-derivation + embedding re-compute.
13. **Failure Handling**: LLM timeout / repeat-loop / NSFW moderation / hostile-tone heuristic → [대화 재시작] [대화 닫기] overlay within 5s. No silent retry, no infinite spinner.
14. **Verified Human Badge**: Discreet World ID Orb badge on every profile/report/match screen.
15. **Cost-cap kill switch**: Daily LLM-spend ceiling triggers automatic streaming pause + user-facing "오늘은 산책이 모두 마감되었어요".

## ADR (Architecture Decision Record)

**Decision**: Build v0 MVP per Option A — Track A end-to-end with Track B as Daily Stroll cards stub, **bilingual schema + i18n shell in Phase 1 with EN prompt rendering deferred to Phase 5 behind `BILINGUAL_PROMPTS_V1` flag**, S-grade investment concentrated in Phase 2 (LLM prompts + streaming infra) with 2-reviewer rubric gate.

**Drivers**:

1. Conversation chemistry is S-grade; everything else is B-grade-OK (spec R9).
2. 4-6 week budget cannot ship dual-track at full polish.
3. Re-engineering bilingual schema post-launch costs more than wiring the shell now; sequencing EN prompt rendering to Phase 5 protects Phase 2 rubric focus without forfeiting bilingual launch posture.

**Alternatives Considered**:

- Option B (both tracks half-quality): rejected — erodes S-grade.
- Option C (KR-only entirely, bilingual deferred): rejected — contradicts R15; re-engineering schema cost higher than upfront. Synthesized variant (KR-only schema + EN entirely v1+) rejected as too far from spec; instead the Critic-suggested compromise is adopted.
- Origin-blind matches UNIQUE: rejected — blocks dual-track pending matches; revised to `(user_id, candidate_user_id, origin)`.
- RLS policy on `conversations` for client SELECT: rejected — opens read surface; instead added `/api/conversation/[id]/stream` to `getServiceClient()` whitelist.

**Why Chosen**: Maximum S-grade preservation within budget; minimum re-engineering risk; preserves all six core schema tables and RLS contract from `0001_initial.sql`/`0002_rls.sql` (additive only); explicit launch-gate on EN prompts so a failed rubric does not hold the launch.

**Consequences**:

- Track B Pokemon-encounter world deferred to v1+. Product owner sees a card stack, not a world, on Day 1.
- All LLM prompts must accept `language` from day 1 — slight prompt-engineering overhead.
- Streaming infra is now a load-bearing dependency (can't be downgraded without erasing the live-rendering acceptance criterion).
- Two distinct rubric passes (KR in Phase 2, EN in Phase 5.5) — 2 person-days of additional human-eval cost.
- Cost-cap kill switch is now infrastructure: requires PostHog query API access from Inngest; an outage in PostHog could disable the kill switch (acceptable — soft cap, not hard cap).
- `matches.origin`-aware partial UNIQUE permits one match per origin track per pair — explicit design choice; reports surfaced to user are filtered by origin in `/api/match/...`.

**Follow-ups**:

- Schedule v1 Sims-world track 2-3 weeks post-launch.
- Cross-language matching design pass at v1.
- WLD payments + Agora ritual remain v2 per existing roadmap.
- Avatar regeneration policy revisit at v1 if user complaints surface.
- EN rubric re-eval at v1.1 if Phase 5.5 fails.
- Cost-cap to be replaced with hard rate-limit at LLM gateway in v1.
